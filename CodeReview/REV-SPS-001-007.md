## Downstream finding closure

### Canonical plan ordering

The manifest, schemas, public documentation, PowerShell planner, semantic
validator, tests, examples and committed expected plans consistently require
`migration-first-manifest-order-v1`.

- A blocked plan contains exactly one sequence-1 `block` action.
- Deterministic migration actions precede ordinary manifest actions.
- Version 1 has one possible AGENTS migration; future migrations must use their
  declared migration-rule order.
- Ordinary entries follow the exact selected manifest array order. Clients may
  not privately sort by path, ID, ownership, source or action.
- A required backup immediately precedes its identity-equal mutation.
- Sequence is assigned last and is contiguous from 1.
- Repeated planning against unchanged state is semantically identical.

### AGENTS collision, idempotency and races

The implementation hashes the exact source bytes and uses
`AGENTS-project.migration-sha256-<lowercase-sha256>.md`. An absent destination
emits one sequence-1 migration. An identical regular file emits no migration.
Different bytes and unsupported objects fail before mutation with stable
failure classes. Apply rechecks the source and destination, uses exclusive
creation, never chooses a second path and completes or rejects the migration
before ordinary entry mutations.

The focused candidate fix preserves physical-containment checks while exposing
`agents-migration-destination-unsupported-object` for a deterministic
preservation-target link or reparse object.

### Shared language-neutral conformance package

`Toolkit/conformance/install-v1/` contains a portable schema-validated scenario
index and 17 authoritative expected outcomes: 11 applicable plans, 3 blocked
plans and 3 fatal outcomes. Every scenario uses `archive-no-git`. Normal
validation and CI compare committed authorities and never regenerate them;
candidate generation remains an explicit maintainer-only operation.
