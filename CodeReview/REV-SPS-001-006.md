## Findings

### Medium 1 — normative action ordering is missing

The public contract states that plans are ordered and sequential, but does not
normatively define migration precedence, manifest-entry array order, adjacency
of backup/mutation pairs or prohibition of private client sorting. PowerShell
behavior cannot serve as the only definition of canonical `PlanJson`.

Required remediation: publish and machine-enforce one canonical v1 ordering
algorithm covering blocked plans, migrations, manifest order, adjacent
backup/mutation pairs, final contiguous sequence assignment and deterministic
repeat planning.

### Medium 2 — AGENTS conflict collision and idempotency rules are incomplete

The deterministic `AGENTS-project.migration-sha256-<content-hash>.md`
destination lacks complete public semantics for an absent target, an identical
regular file, a different regular file, an unsupported filesystem object and a
target that appears or changes between plan and apply.

Required remediation: hash exact source bytes, define the canonical no-action
idempotent case, use a stable fail-closed pre-plan failure model for collisions
and invalid objects, recheck source/destination assumptions immediately before
migration apply, and never overwrite or redirect project-owned preservation
content.

### Medium 3 — the shared conformance package is absent

Implementation-specific PowerShell tests and one illustrative plan do not let
an independent Go client prove equivalent behavior for empty, legacy, upgrade,
force, initialize, repeat-initialize, archive and error scenarios.

Required remediation: check in a versioned language-neutral conformance package
with portable before states, options, authoritative normalized expected plans or
stable expected fatal-failure classes, preservation assertions, validation and
a reference harness. Normal CI must compare against reviewed committed
expectations rather than regenerate them from the implementation under test.
