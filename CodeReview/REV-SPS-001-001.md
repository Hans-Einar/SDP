## Disposition

Changes required. There were no blocking findings, but unresolved high and
medium findings prevent Slice completion and draft-PR publication. The passing
local verification remains valid evidence for the exact implementation commit;
it is not evidence that the reviewed contract is safe or compatible enough to
advance.

## High findings

### H1 — Physical path containment can be bypassed

`Toolkit/scripts/Install-SDP.ps1` uses lexical `GetFullPath` containment for the
project root and destinations. Existing Windows junctions (and equivalent
symlinks) are not resolved. The Reviewer reproduced a plan that targeted the SDP
source through a junction and an apply that wrote outside the intended project
root.

Required remediation: resolve or reject reparse points/symlinks for the project
root and every existing destination ancestor, reject physically resolved
source/target escapes before mutation, and add Windows-junction plus applicable
Unix-symlink regressions.

### H2 — YAML scalar scanning bypasses schema and downgrade gates

`Get-YamlScalar` selects matching keys at any indentation. A manifest with
nested supported values before unsupported root `schemaVersion` and newer root
`toolkitVersion` values produced `canApply: true`, bypassing both unsupported
schema and downgrade protection. Malformed YAML containing the expected scalar
text was also accepted.

Required remediation: parse and validate the root mapping with a real or strict
supported-subset parser, reject duplicate/nested-shadow/malformed required
fields before target mutation, and add nested-shadow and malformed fixtures for
both installed and project manifests.

### H3 — PowerShell does not enforce the complete v1 manifest contract

The installer manually checks only a subset of the JSON Schema. The Reviewer
confirmed that unknown policy fields and schema-invalid exclusion/source-class
changes could be ignored while an applicable mutating plan was emitted.

Required remediation: implement schema-equivalent closed-world validation for
all v1 properties, required fields, array minima, policy fields and
exclusion/source-class invariants before target inspection; add adversarial
PowerShell fixtures proving invalid contracts fail without mutation.

### H4 — Portable paths permit Windows aliases and sensitive destinations

The schemas and runtime accept segments with trailing dots/spaces, reserved
device names and other Windows-invalid aliases. A source ending in a trailing
dot bypassed the raw exclusion for `RELEASE-NOTES.md` while Windows resolved it
to the excluded live file. `.git` destinations are also not prohibited.

Required remediation: define one cross-platform segment subset, reject
Windows-equivalent aliases, control characters, reserved device names, ADS/
colon forms and `.git` destinations, compare normalized/physical paths for
exclusion and uniqueness, and cover every source/destination/governing/
exclusion path class.

### H5 — The exact motivating `gh-sdp` project does not validate

Project mode rejected the pinned closure head
`ed205c1ef193ab8a6e5cd1c50e558c3049ce6def`. Incompatibilities include the
existing `release.targetStatus`, project relation categories, list-valued Slice
release links and project-root-relative `SDP/...` paths being resolved as
`SDP/SDP/...`. The installer preserves those project-owned records and provides
no migration or warning, so an installed/upgraded motivating consumer remains
unvalidateable.

Required remediation: define a compatible path-base rule, preserve or explicitly
namespace supported project extensions, document the compatibility boundary and
add the exact exported `gh-sdp` head as a project-mode regression fixture.

## Medium findings

### M1 — AGENTS migration plan differs from apply-time I/O

Conflict plans identify the template or `AGENTS-project.md`, but apply copies the
existing managed `AGENTS.md` to a timestamped migration path. Model target-to-
target migration explicitly in the plan schema and make apply execute the
represented action; add plan-versus-apply conflict fixtures.

### M2 — PowerShell version ordering is not full SemVer

The comparer discards prerelease/build identifiers while schemas accept them.
Implement SemVer 2.0 precedence and defined build-identity behavior, or narrow
the accepted public versions consistently; add final/prerelease downgrade cases.

### M3 — Generated installed-manifest equivalence is serialization-dependent

PowerShell emits one YAML serialization but compares byte-for-byte, so an
independent equivalent serializer can produce different plans or refresh churn.
Declare canonical serialization or compare generated facts semantically and add
a cross-language golden/equivalence fixture.

### M4 — Plan reason vocabulary is not normative

The schema permits any stable reason ID while PowerShell relies on an
undocumented vocabulary. Enumerate stable reasons and their decision conditions
in the schema or normative decision table so independent clients can produce
equivalent plans.

### M5 — Review/verification and reverse traceability links are not checked

The validator omits `REV-*` and `VER-*` Ledger subject mapping and does not
validate review, verification and migration reverse links. The Reviewer removed
or corrupted those links and still received no errors. Validate all known
relation fields and add dangling review/verification/reverse-link regressions.

### M6 — `.yml` Release and Fix records can evade validation

Relation paths accept `.yml`, while discovery scans only `*.yaml`. A malformed
referenced release renamed to `.yml` passed project validation. Either validate
both extensions or require canonical `.yaml` consistently, with Release and Fix
regressions.

### M7 — Publication identity/state contradictions can pass

Unreleased records can contain non-null publication fields, and release-event
tags need not match their release ID. Add state-specific publication nullability,
record/Relations identity agreement and tag/version/release-ID semantic checks.

### M8 — Plan schema alone accepts contradictory action sequences

Duplicate/non-contiguous sequence numbers and action versions that contradict
the top-level version validate. Add semantic plan validation for ordered unique
sequences, version agreement, and manifest entry/source/generator/destination
agreement, with adversarial tests.

### M9 — Public SemVer patterns accept invalid identifiers

Several schemas accept numeric prerelease leading zeros, consecutive dots and
empty identifiers that the implementation otherwise treats as invalid. Apply a
SemVer-2.0-correct rule consistently to every version-bearing boundary and add
schema-level negative cases.

### M10 — Governing schema/capability pairings can drift

Three manifest entries pair CurrentIndex, Relations and Ledger schemas with the
wrong capabilities, and validation only checks that each capability exists
somewhere. Correct the pairings and assert canonical schema-to-capability
relationships.
